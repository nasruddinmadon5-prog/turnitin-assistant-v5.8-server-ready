# v5.2 UI Alignment Changes

## Overview
This version fixes the UI/dashboard alignment to reflect the real backend workflow state instead of dummy/static HTML. The changes focus on making the dashboard dynamic and responsive to actual log messages from the backend engines.

## Behavior Changes

### Before (v5.1)
- System Output panel expanded endlessly with logs (no scroll limit)
- Pipeline visualization was static dummy HTML
- System Status cards showed hardcoded "Ready" text
- Version displayed as "v5.0 Bridge"
- No visual feedback when workflows started

### After (v5.2)
- System Output panel has fixed 400px height with internal scrollbar
- Pipeline nodes update dynamically based on log patterns:
  - **OJS Login:** activates when logs contain "Launching browser with persistent OJS profile", "Please login manually", "Already logged in", or "Login detected"
  - **Download Article:** activates with "PROCESS:", "[DOWNLOAD]", "[SUCCESS]", "Downloaded:"
  - **Excel Bridge:** activates with "[EXCEL]", "Sudah downloaded dalam Excel", "Logged to turnitin", "data/master update.xlsx"
  - **Turnitin Upload:** activates with "[TURNITIN] Starting upload workflow", "[BRIDGE] Selected submission_no", "UPLOAD_FORM_FILLED", "UPLOADED", "CONFIRMED"
  - **Similarity Filter:** activates with "Opening All Sources", "Filter Sources", "FINAL SCORE AFTER FILTERING", "FILTER_DONE"
  - **Download Report:** activates with "Download PDF", "DOWNLOADED", "PDF saved", "Report:"
- System Status cards update dynamically:
  - **OJS ENGINE:** Ready → Running → Waiting Login → Completed → Failed
  - **TURNITIN ENGINE:** Ready → Running → Waiting Login → Filtering → Downloading → Completed → Failed
  - **EXCEL BRIDGE:** Shows file path + status (Ready → Reading → Updated → Error)
- Version displays as "v5.2 Bridge"
- Visual feedback: nodes glow and change colors as workflow progresses

## Assumptions

1. **Log Message Patterns:** The UI relies on specific log message patterns emitted by the backend engines. The patterns are based on existing log messages found in the codebase:
   - OJS engine logs: "Launching browser", "Please login manually", "Login detected"
   - Turnitin engine logs: "[TURNITIN]", "UPLOADED", "CONFIRMED", "FILTER_DONE"
   - Excel bridge logs: "[EXCEL]", "data/master update.xlsx"

2. **Log Callback Compatibility:** The backend already calls `addLog(message, 'INFO')` through the JavaScript bridge, which matches the UI function signature.

3. **Single-Window Operation:** The UI assumes the dashboard remains open during workflow execution (pywebview window stays active).

4. **No Backend State API:** The solution uses frontend-only state management by parsing log messages. No new backend API endpoints were added.

5. **Sequential Workflow:** The pipeline visualization assumes the workflow follows the sequence: OJS Login → Download → Excel Bridge → Turnitin Upload → Similarity Filter → Download Report.

   - Pipeline "OJS Login" node lights up when browser launches
   - System Status "OJS ENGINE" changes from Ready → Running

   - System Status updates accordingly
5. Verify no separate login buttons are present
6. Verify window remains usable at 1400x900 resolution

### Known Limitations
- Pipeline state is inferred from log messages, not from actual backend state
- If log messages change format in the future, UI patterns may need updating
- Error states are detected by presence of "ERROR", "FAILED", or "❌" in logs
- Full Workflow button is KIV (kept in view) as per v5.1

## Migration from v5.1

### For Users
- Simply run `python3 app.py` from the v5.2 directory
- No configuration changes required
- Existing profiles and data are preserved (copied during duplication)

### For Developers
- The UI now expects log messages to follow existing patterns
- If adding new workflow stages, update the `updatePipelineFromLog()` function in index.html
- If adding new engine states, update the `updateSystemStatusFromLog()` function in index.html

## Summary

This release focuses purely on UI/UX improvements without touching the core engine logic. The dashboard now provides real-time visual feedback of workflow progress, making it easier to monitor automation status. All changes are frontend-only, maintaining full compatibility with the existing backend architecture.
